# Поиск в мобильном приложении банка

## Описание сценария

Пользователь мобильного приложения банка нажимает на строку поиска и вводит текст запроса (например, «перевод по
номеру»). Система выполняет поиск одновременно по нескольким источникам: по внутренним экранам (разделам) приложения и
по истории операций клиента через внешний сервис. Результаты агрегируются и возвращаются пользователю в едином ответе.

Система поддерживает историю поисковых запросов клиента (последние 10 запросов), которая используется на экране
подсказок (эндпоинт /suggested).

---

## Эндпоинт

**POST** `/api/v1/search`

Content-Type: `application/json`

---

## Входящие параметры

Тело запроса оборачивается в стандартную обёртку `CommonRequest`, данные передаются в поле `data`:

| Параметр      | Тип            | Обязательный | Описание                                                    |
|---------------|----------------|--------------|-------------------------------------------------------------|
| clientId      | String         | Да           | Уникальный идентификатор клиента                            |
| searchString  | String         | Да           | Текст поискового запроса. Минимальная длина 3 символа       |
| platform      | String         | Да           | Платформа клиента: `ios`, `android`, `web`                  |
| clientVersion | String         | Нет          | Версия приложения клиента (например, `1.22.3`)              |
| searchTypes   | List\<String\> | Нет          | Типы поиска: `PAGE` (разделы), `HISTORY` (история операций) |
| size          | Integer        | Нет          | Максимальное количество результатов (по умолчанию 10)       |

**Пример запроса:**

```json
{
  "data": {
    "clientId": "9fb5188b-35d5-4a80-994f-0d158bd260bd",
    "searchString": "перевод по номеру",
    "platform": "ios",
    "clientVersion": "1.22.3",
    "searchTypes": [
      "PAGE",
      "HISTORY"
    ],
    "size": 10
  }
}
```

---

## Исходящие параметры

Ответ оборачивается в стандартную обёртку `CommonResponse`. Поле `data` содержит:

| Параметр        | Тип                        | Описание                                 |
|-----------------|----------------------------|------------------------------------------|
| section         | List\<SearchSectionDto\>   | Результаты поиска по разделам приложения |
| externalSection | List\<ExternalSectionDto\> | Результаты поиска из внешних источников  |

### SearchSectionDto (разделы приложения)

| Поле       | Тип             | Описание                    |
|------------|-----------------|-----------------------------|
| entityType | String          | Всегда `"SECTION"`          |
| entityName | String          | Всегда `"Разделы"`          |
| pages      | List\<PageDto\> | Найденные экраны приложения |

Каждый `PageDto` содержит: `name`, `link`, `icon`, `featureToggles`.

### ExternalSectionDto (внешние источники)

| Поле       | Тип                         | Описание                                           |
|------------|-----------------------------|----------------------------------------------------|
| entityType | String                      | `"OPERATION"` или `"HISTORY"`                      |
| records    | List\<OperationRecordDto\>  | Записи операций (для типа `OPERATION`, иначе null) |
| operations | List\<HistoryOperationDto\> | История операций (для типа `HISTORY`, иначе null)  |

**OperationRecordDto:** `id`, `name`, `categoryCode`, `typeCode`, `icon`

**HistoryOperationDto:** `id`, `name`, `amount` (BigDecimal), `currency`, `date` (ISO 8601)

**Пример ответа:**

```json
{
  "status": "SUCCESS",
  "actualTimestamp": "2024-05-15T12:00:00.000Z",
  "data": {
    "section": [
      {
        "entityType": "SECTION",
        "entityName": "Разделы",
        "pages": [
          {
            "name": "Перевод по номеру телефона",
            "link": "https://host/transfer/phone",
            "icon": "ic_transfer",
            "featureToggles": []
          }
        ]
      }
    ],
    "externalSection": [
      {
        "entityType": "OPERATION",
        "records": [
          {
            "id": "pay_123",
            "name": "Оплата мобильной связи",
            "categoryCode": "COMMUNICATION",
            "typeCode": "MOBILE",
            "icon": "ic_mobile"
          }
        ]
      },
      {
        "entityType": "HISTORY",
        "operations": [
          {
            "id": "txn_890",
            "name": "Перевод Олегу Т.",
            "amount": -500.0,
            "currency": "RUB",
            "date": "2024-05-14T18:30:00Z"
          }
        ]
      }
    ]
  }
}
```

## Алгоритм работы

### 1. Параллельный поиск по источникам

Поиск по разным источникам выполняется параллельно:

- **Поиск по разделам** (если `searchTypes` содержит `PAGE`):
    - Выполняется SQL-запрос к таблице `pages` с использованием триграммного сходства по полю `dictionary`. Порог similarity = 0.5
    - Результаты фильтруются по платформе (`platform`) и версии приложения (`clientVersion`)
    - Результаты сортируются по убыванию релевантности
    - Найденные страницы маппятся в `SearchSectionDto`

- **Поиск по истории операций** (если `searchTypes` в запросе содержит `HISTORY`):
    - Отправляется HTTP-запрос во внешний сервис `operation-history`
    - При недоступности внешнего сервиса возвращается пустой результат, ошибка логируется
    - Результаты маппятся в `ExternalSectionDto`
    - Для тестирования можно запустить wiremock сервер по инструкции из /mock/operation-history/README.md

### 2. Сохранение в историю поиска

Выполняется асинхронно, не блокирует ответ пользователю:

- Запрос поиска сохраняется в таблицу `latest_results` с привязкой к `clientId`
- Если у клиента накопилось более 10 записей, самые старые удаляются (ротация)

### 4. Формирование ответа

Результаты из всех источников агрегируются в единый объект `SearchResponseData` и оборачиваются в `CommonResponse`.

## Интеграция с внешним сервисом

| Параметр     | Значение                                              |
|--------------|-------------------------------------------------------|
| Сервис       | `operation-history`                                   |
| Протокол     | HTTP (REST)                                           |
| Клиент       | `RestClient` (Spring 6.1+)                            |
| Конфигурация | `integration.operation-history.url` в application.yml |
| Деградация   | При ошибке — пустой результат, лог ошибки             |
